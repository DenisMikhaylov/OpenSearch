# Лабораторная работа: Мониторинг состояния кластера OpenSearch с использованием OpenSearch Dashboards


## Среда выполнения

*   **ОС хоста:** Debian 13.
*   **Контейнеризация:** Docker и Docker Compose.
*   **Стек:**
    *   **OpenSearch** (образ `opensearchproject/opensearch:latest`) — кластер для хранения данных.
    *   **OpenSearch Dashboards** (образ `opensearchproject/opensearch-dashboards:latest`) — интерфейс для визуализации и мониторинга.
    *   **OpenSearch UI** — доступен через Dashboards начиная с версии 2.17 для Cluster Insights .

> **Важно:** Cluster Insights в OpenSearch Dashboards доступен для версий OpenSearch **2.17 и выше**, а Query View — для версий **2.19 и выше** .

## Часть 2. Знакомство с Cluster Insights

**Cluster Insights** — это единая панель мониторинга, которая предоставляет курируемые аналитические данные и рекомендации по устранению проблем .

### Шаг 1. Доступ к Cluster Insights

1. Откройте браузер и перейдите по адресу `http://<IP_вашего_сервера>:5601`.
2. Авторизуйтесь как `admin` с паролем `MyStrongPassword123!`.
3. Нажмите на иконку **Settings** (шестерёнка) в левом нижнем углу экрана .
4. Выберите **Data administration** > **Cluster Insights** .

### Шаг 2. Изучение Overview Dashboard

Панель **Cluster Insights Overview** содержит следующие разделы :

| Раздел | Описание |
|--------|----------|
| **Current cluster status** | Круговая диаграмма статуса кластера (Green/Yellow/Red)  |
| **Insights trend** | График тренда проблем за последние 30 дней  |
| **Current open insights** | Количество активных инсайтов с разбивкой по серьёзности  |
| **OpenSearch Service Clusters** | Список кластеров с ключевыми метриками  |
| **Top insights by severity** | Приоритизированные проблемы с описанием и рекомендациями  |

## Часть 3. Детальный мониторинг кластера

### Шаг 1. Cluster Health View

Выберите ваш кластер в разделе **OpenSearch Service Clusters**. Откроется страница с детальной информацией :

**Overview:**

*   **Ключевые метрики:** статус кластера, количество шардов, узлов, индексов и документов .
*   **Configuration best practices:** круговые диаграммы, показывающие соответствие рекомендациям по резервированию и безопасности .

**Insights:**

Таблица с активными инсайтами для данного кластера. Каждый инсайт содержит :
*   Описание проблемы
*   Список затронутых ресурсов (индексы, шарды)
*   Пошаговые рекомендации по устранению
*   Историю исправлений

**Metrics Section:**

Интерактивные графики, отображающие :
*   Общие метрики здоровья кластера (статус, статус записи)
*   KPI (скорость и задержки индексации/поиска)
*   Использование ресурсов (JVM, CPU)

## Часть 4. Детальные представления: Node, Index, Shard, Query

### Шаг 1. Node View (версия 2.17+)

Вкладка **Node View** показывает производительность каждого узла :

**Ключевые метрики:**
*   Heat Score — общий показатель здоровья узла
*   Использование ресурсов (CPU, память, диск)
*   Задержки и скорость поиска/индексации
*   Быстрые ссылки на Top N шардов и запросов на узле

**Практическое задание:**

1. Перейдите на вкладку **Nodes view**.
2. Определите узел с наибольшей загрузкой CPU.
3. Нажмите на ID узла для просмотра временных графиков ресурсов.
4. Нажмите на ссылку **Top N shards**, чтобы увидеть шарды, работающие на этом узле .

### Шаг 2. Index View

Вкладка **Index View** показывает метрики на уровне индексов :

| Метрика | Описание |
|---------|----------|
| Document count | Количество документов в индексе |
| Store size | Размер хранилища |
| Search latency/rate | Задержка и скорость поиска |
| Indexing latency/rate | Задержка и скорость индексации |
| Top N queries | Ссылка на ресурсоёмкие запросы для индекса |

**Практическое задание:**

1. Перейдите на вкладку **Index view**.
2. Найдите индекс с наибольшим размером (`store size`).
3. Определите индекс с самой высокой задержкой поиска.
4. Нажмите на индекс для просмотра детализированных метрик.

### Шаг 3. Shard View (самый детальный уровень)

Вкладка **Shard View** отображает метрики для каждого отдельного шарда :

*   Shard ID и его узел
*   Resource pressure (CPU, memory)
*   Search/Indexing latency на шард

**Практическое задание:**

1. Перейдите на вкладку **Shard view**.
2. Найдите шард с наибольшим потреблением heap (`Top shard heap allocated`).
3. Определите шард с самой высокой задержкой поиска.
4. Используйте эту информацию для выявления "горячих" шардов.

### Шаг 4. Query View (версия 2.19+)

Вкладка **Query View** помогает мониторить ресурсоёмкие запросы :

**Live dashboards:**

*   Статистика выполнения запросов в реальном времени
*   Использование CPU и памяти
*   Прогресс выполнения

**Top N queries:**

Таблица ранжированных запросов с информацией :

| Поле | Описание |
|------|----------|
| Query count | Количество выполнений запроса |
| Latency | Задержка выполнения |
| CPU/Memory usage | Использование ресурсов |
| Search type | Тип поиска (query, fetch, expand) |
| Coordinator node | Узел-координатор |
| Target indices/shards | Целевые индексы и шарды |

**Практическое задание:**

1. Перейдите на вкладку **Query view** (если версия 2.19+).
2. Выполните несколько поисковых запросов через Dev Tools.
3. Вернитесь в Query View и найдите свои запросы в списке.
4. Дважды кликните на любой запрос, чтобы увидеть:
    *   Точный payload запроса
    *   Разбивку задержки по фазам (expand, query, fetch) 
    *   Рекомендации по оптимизации

### Просмотр метрик через Dashboards

После включения мониторинга :

1. Откройте **OpenSearch Dashboards** → **Dashboard**.
2. Выберите панель **[Monitoring] Index monitoring Dashboard**.
3. Доступные графики:

| График | Описание |
|--------|----------|
| Indexing Rate (/s) | Скорость записи документов |
| Search Rate (/s) | Скорость запросов |
| Top 10 индексов по записи | Индексы с наибольшей скоростью записи |
| Top 10 индексов по запросам | Индексы с наибольшей скоростью запросов |
| Total docs count | Общее количество документов |
| Total store size | Общий размер хранилища |
| Top 10 по размеру | Индексы с наибольшим размером |
| Top 10 по сегментам | Индексы с наибольшим количеством сегментов |
